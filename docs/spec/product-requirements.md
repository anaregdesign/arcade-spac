# Arcade App 要件整理

## Summary

複数の arcade game を 1 つのアカウントと共通 UI で横断し、大量のゲーム群から迷わず選んで遊び、各ゲームの上達や成績確認へ滑らかに進める組織内向けゲームポータルを定義する。

## User Problem

- 単体ゲームごとに遊び場や成績確認の導線が分かれると、継続利用しづらい
- 生のゲーム結果だけでは、複数ゲームを横断した総合的な実力が見えにくい
- 競争要素があっても、自分の現在地、次に伸ばすべきゲーム、結果の共有先が分断されると体験が弱くなる

## Scope

- `Microsoft Entra ID` ログインを前提にした組織内ユーザ向けゲームポータル
- `マインスイーパ`、`数独`、`Drop Line` の 3 ゲーム
- ホーム、ゲーム、結果、ランキング、プロフィール、ヘルプの主要体験
- ゲーム別ランキング、総合ランキング、総合得点、プレイ履歴、`Microsoft Teams` 共有

## Non-Goals

- フレンド機能やチャットなどのソーシャル機能
- リアルタイム対戦やチーム戦
- デイリー、ウィークリーなど MVP 外の追加ランキング期間
- マルチテナント対応

## Acceptance Criteria

- ユーザは `Microsoft Entra ID` でサインインし、初回利用時でもホーム画面上の強調表示されたゲーム候補から、そのまま目的のゲームを選んで開始できる
- ユーザはホーム画面、ゲーム画面、結果画面のいずれからでも次の行動を判断でき、ゲーム間を迷わず切り替えられる
- ユーザはホーム画面をゲーム選択専用ハブとして利用でき、100 個から 200 個規模のゲームが増えても検索、絞り込み、分類、並び替えで目的のゲームへ素早く到達できる
- ユーザは主要操作と現在の状態を初期表示の範囲で把握でき、過度なスクロールなしでプレイ開始、再挑戦、ゲーム切り替えを行える
- プレイ結果画面では、今回の記録、自己ベスト差分、ランキングまたは総合得点への影響、`集計待ち` や `保存保留` などの状態、再挑戦と共有導線を共通レイアウトで確認できる
- ランキング画面では、`総合` と `ゲーム別`、`シーズン` と `累計` を切り替えても自分の順位と差分を見失わない
- プロフィール画面では、ゲーム別ベスト、総合得点の内訳、成長推移、表示名と公開範囲を確認または変更できる
- プロフィール画面では、ゲーム別ベスト、総合得点の内訳、成長推移、表示名、公開範囲、表示テーマを確認または変更できる
- 進行中プレイから離脱する場合、共有リンクを未認証で開く場合、結果保存が失敗する場合のいずれでも、ユーザは次に取るべき行動を画面上で理解できる

## Constraints and Dependencies

- MVP の認証対象は `AzureADMyOrg` を前提とした同一テナント内ユーザに限定する
- 共有リンク、ランキング詳細、プロフィール閲覧は同一テナントの認証済みユーザだけが利用できる
- スコア登録、ランキング集計、プロフィール保持を扱うため、静的配信だけではなくサーバランタイムを前提にする
- MVP の対象ゲームは 3 つ、対象期間は `シーズン` と `累計` に限定する

## 1. このアプリが提供する体験

このアプリは、1 人のユーザが複数の arcade game を 1 つのアカウントで横断して遊び、各ゲームの成績と総合成績の両方で他ユーザと競えるゲームポータルである。

初期対象ゲームは次を想定する。

- マインスイーパ
- 数独
- Drop Line

ユーザは `Microsoft Entra ID` でログインし、ホーム画面で大量のゲーム候補から目的のタイトルを選び、各ゲーム画面や結果画面からも別ゲームへスムーズに移動できる。

このアプリの価値は次の 3 点にある。

- 複数ゲームを 1 つの導線で遊べること
- ゲーム別の上達と総合的な強さを両方見られること
- 他ユーザとの比較結果がわかりやすく可視化されること

## 2. 想定ユーザ

### 2.1 メインユーザ

- 日常的に短時間でゲームを遊びたいユーザ
- 同じゲームを繰り返し練習し、ランキング上位を狙いたいユーザ
- 1 つのゲームに偏らず、複数ゲームを横断して総合力を高めたいユーザ

### 2.2 ユーザの期待

- ログイン後すぐに何で遊ぶか選べること
- ゲーム数が大幅に増えても、目的のゲームを短時間で見つけられること
- ゲームカテゴリや状態で自然に候補を絞り込めること
- 1 回ごとの結果だけでなく、継続的な成長が見えること
- 他人と比べてどの位置にいるかがすぐわかること
- ゲームを変えるたびにアプリの文脈が切れないこと

## 3. 主要ユースケース

### 3.1 初回利用

1. ユーザはログイン画面を開く。
2. `Microsoft Entra ID` でサインインする。
3. 初回ログイン時はプロフィールの初期作成が行われる。
4. ログイン直後にホーム画面兼ゲーム選択画面へ遷移する。
5. 初回利用時でもホーム画面本文はゲーム選択 UI のみを表示し、推奨ゲームの強調表示から最初の 1 本を選べるようにする。
6. ユーザは一覧または検索から目的のゲームを選んでプレイを開始する。

### 3.2 日常利用

1. ユーザはログインする。
2. ホーム画面でカテゴリ、検索、並び替え、強調表示を使って遊びたいゲームを見つける。
3. 遊びたいゲームを選ぶ。
4. プレイ結果を確認する。
5. 結果画面またはゲーム内ナビゲーションから別ゲームへ移動する。
6. その日のうちに複数ゲームを切り替えながら遊ぶ。

### 3.3 競争利用

1. ユーザはあるゲームのランキングを見る。
2. 自分より上位のユーザとの差分を確認する。
3. 自己ベスト更新を狙って再挑戦する。
4. 必要に応じて別ゲームへ移動し、総合得点の底上げを狙う。
5. 総合ランキング画面で、複数ゲームの総合力としての自分の順位を確認する。

## 4. 画面とユーザ行動の整理

### 4.1 ログイン画面

#### ユーザがやりたいこと

- 安全にログインしたい
- ログイン後に何ができるかを理解したい

#### 必須要件

- `Microsoft Entra ID` によるサインイン導線を提供する
- 未ログイン状態ではランキングの詳細閲覧やスコア登録を制限する
- ログイン成功後はホーム画面へ遷移する
- ログイン失敗時は理由がわかるメッセージを表示する

#### 表示要素

- アプリ名と短い説明
- サインインボタン
- 未ログイン状態で見られる範囲の説明
- 利用規約やプライバシーへの導線

#### 追加で必要な要件

- 前回ログイン済みのユーザには、不要な再ログインを避けて自然にホームへ復帰できること
- 初回利用者でも、ホーム画面上に追加案内を差し込まず最初のゲームを選べること

### 4.2 ホーム画面兼ゲーム選択画面

#### ユーザがやりたいこと

- 次に遊ぶゲームを選びたい
- 大量のゲーム候補から目的のゲームを素早く見つけたい
- ゲーム数が増えても一覧を迷わず探索したい

#### 必須要件

- ログイン直後のランディング画面とする
- ホーム画面はゲーム選択専用画面とし、主目的をゲーム探索と起動に限定する
- プレイ可能なゲームを大量配置前提の一覧またはグリッドで表示する
- 各ゲームカードから直接ゲーム画面へ遷移できる
- 各ゲームカードは、そのゲームを代表する実ゲーム画面のスクリーンショットを主役として表示し、視覚的に内容を判断できるようにする
- 100 個から 200 個規模のゲーム数に増えても、検索、カテゴリ、タグ、並び替え、ページネーションまたは遅延読み込みで探索性を維持できる
- 各ゲームカードは一覧密度を優先し、起動判断に必要な最小情報だけを表示する
- 各ゲームカードの本文テキストは、ゲーム名と最小限の状態ラベルを中心に抑え、段落説明に依存しないこと
- ホーム画面本文は、ゲーム一覧、探索導線、推奨ゲームの強調表示など、ゲーム選択と起動に直接関係する UI のみで構成する
- 総合得点、ランキング、詳細プロフィールなどの情報はホームの主役にせず、必要時に別画面へ遷移して確認できること

#### 表示要素

- 各ゲームのスクリーンショット preview を含むカード
- 各ゲームのカード
- 検索入力
- カテゴリまたはタグによる絞り込み
- 並び替え導線
- ヘルプを明示的に開くための導線
- 必要に応じたセクション見出し: 新着、人気、最近遊んだ、カテゴリ別 など

#### 追加で必要な要件

- まだ一度もプレイしていないユーザ向けに、空状態でも次の行動がわかること
- 初回利用時は、どのゲームから始めるべきか迷わない導線を、ホーム上の強調表示だけで出すこと
- プレイ履歴が少ない段階でも、ホームの情報不足で迷わずゲーム選択に進めること
- ゲーム数が増えた場合でも、ユーザが 2 から 3 操作程度で目的のゲームへ到達できる情報設計を優先すること
- ヘルプやスコア説明は通常時にホームへ常設表示せず、ユーザが必要なときだけ開けること
- 総合得点、総合順位、最近の結果などの補助情報をホーム本文に置かないこと
- 通常利用時のホーム上部では、ゲーム一覧またはグリッドと探索導線が最初に視認できること

#### レイアウト方針

- ホーム画面の最外周余白は必要最小限に留め、初期表示でより多くのゲームカードを見せること
- セクション間、カード間、見出し下の縦余白は、可読性を損なわない範囲で最小化し、一覧密度を優先すること
- 大きな画面では横幅を活かした多列グリッドまたは高密度リストを使い、左右に広い空白帯を残さないこと
- モバイルではタップしやすさを維持しつつ、1 画面あたりに表示できるゲーム数を優先すること
- ホーム初期表示では説明やサマリーよりもゲーム一覧の露出面積を最優先し、ヘルプ本文を常時差し込まないこと
- ホーム上部の第一ブロックは、ゲーム探索と起動に直接関係する UI だけで構成すること
- 補助的な summary や recent result をホーム本文に表示しないこと
- 各ゲームカードは一覧性を損なわない情報量に抑え、詳細なスコア説明や成長情報はプロフィールや結果画面へ送ること
- 各ゲームカードではスクリーンショット preview が最大面積を占め、テキスト説明は preview を補足する最小限に留めること
- responsive layout でも preview の aspect ratio とカードサイズの整列を保ち、カードごとの高さばらつきで一覧性を損なわないこと

### 4.3 ゲーム画面

#### ユーザがやりたいこと

- いま遊んでいるゲームに集中したい
- 途中で別のゲームへ切り替えたい
- 必要なときだけ操作方法や保存状態を確認したい

#### 必須要件

- ゲーム画面は極力画面全体を利用し、プレイ領域を最大化する
- 現在のゲーム名と難易度を明示する
- 操作方法、保存保留時の扱い、離脱時の扱いは、共通 Help または同一画面内の補助レイヤから即座に確認できること
- 操作説明は本文へ常時露出させず、必要時にだけ明示的に開ける構成でよい
- 各ゲーム画面には、主要 action 群から 1 操作で開ける game-specific 操作ガイド導線を表示すること
- game-specific 操作ガイドは、ゲームごとに内容だけ差し替えつつ、開き方、閉じ方、見出し構造、情報密度は共通 UI パターンで統一すること
- ホームへ戻る導線を常に表示する
- 他ゲームへ直接遷移できるゲーム切り替え導線を表示する
- プレイ終了後は結果画面を表示する
- clear、failed、またはユーザが `Finish run` 相当の終了 action を実行した時点で、追加の `保存` 確認操作を要求せず即座に結果画面へ遷移すること
- touch device でも desktop と同等にプレイ完了できるよう、secondary action が必要なゲームは context menu 以外の first-class な操作手段を持つこと
- 現在の run state で実行できない control は、見た目と操作の両方で `使えない状態` が明確にわかること
- 再挑戦を素早く促す場合に限り、結果モーダルを補助的に使ってよい。ただし情報構成は結果画面と同等にする
- プレイ中断時の扱いを定義する
- 途中でホームや他ゲームへ移動する場合は、進行中プレイが失われるかどうかを事前に明示する
- 進行中プレイからホームや他ゲームへ移動する場合は、確認ダイアログを表示する
- 確認して離脱したプレイは `中断` として履歴に残してよいが、ランキングと総合得点の集計対象外とする
- プレイ完了後に保存失敗した場合は、結果を `保存保留` として保持し、再試行導線と未確定状態を表示する

#### 表示要素

- ゲーム本体
- タイマーまたは進行状況
- 難易度選択
- ホームへの導線
- 他ゲーム切り替えメニュー
- 必要に応じた run 終了 action
- 主要 action 群内の操作ガイド導線
- 必要時に開ける操作方法または recovery help

#### レイアウト方針

- プレイ領域を主役にし、周辺 UI は邪魔にならない最小限の密度で配置する
- 常設の補助カード、自己ベスト差分、ランキング影響 summary など、プレイ判断に必須でない情報を本文へ並べないこと
- run 完了後に `Record current clear` のような追加確定 action を要求せず、結果保存と画面遷移は自動で処理すること
- ゲームごとの core interaction は、そのゲームに自然な 1 手目から始められることが望ましく、不要な開始ステップを増やさないこと
- 操作方法ガイドは共通 Help などの明示的に開く補助レイヤで提供してよい
- 操作方法ガイドは、開いたときだけ `run flow`、`board controls`、`failure or abandon handling` を短く参照できる構成にすること
- モバイルでもデスクトップでも、重要操作と操作方法への到達が同じ画面内または同一ヘッダ導線内で完結すること
- モバイルでもデスクトップでも、操作方法への到達は game screen 上の primary controls から同じ 1 操作で完結すること
- 横向きや広い画面ではプレイ領域を広く使い、狭い画面では説明や補助情報を再配置してもゲーム体験を阻害しないこと

#### 追加で必要な要件

- 一時停止の有無と、そのときタイマーやランキング対象条件がどう変わるかを明示すること
- 通信失敗時や保存失敗時にも、ユーザが現在のプレイ状態を理解できること

### 4.4 プレイ結果画面

#### ユーザがやりたいこと

- 今回の結果が良かったのかすぐ知りたい
- ランキングや総合得点への影響を確認したい
- そのまま再挑戦または別ゲームへ移動したい

#### 必須要件

- 今回の記録を明示する
- 初期表示では今回の記録を最も強い visual hierarchy で表示する
- 全ゲームで共通の結果画面スタイルを用いる
- 自己ベスト更新の有無を表示する
- ゲーム別ランキングの暫定順位変動、またはランキング対象外理由を表示する
- 総合得点の暫定増減を表示する
- 集計確定前は、順位と総合得点が暫定値であることを明示する
- `保存保留` 時は、確定前であることと再試行導線を表示する
- `Microsoft Teams` で結果を共有できるリンクを配置する
- 再プレイと他ゲーム遷移の両方を提供する

#### 表示要素

- 今回スコア
- 自己ベストとの差
- 総合得点の増減または未確定表示
- 順位変動またはランキング対象外表示
- Microsoft Teams 共有リンク
- 再挑戦ボタン
- 他ゲームへ移動するボタン

#### 結果共有の要件

- 共有リンクは結果画面内で視認しやすい位置に配置する
- 共有内容には少なくともゲーム名、今回の結果、自己ベスト更新有無、総合得点または順位変動を含める
- 共有時にゲームごとの表示差は許容するが、レイアウトと導線は共通化する
- 共有導線は再挑戦や他ゲーム移動の主要導線を邪魔しない配置にする
- 共有リンク先は、同一テナントの認証済みユーザのみ閲覧できること
- 未認証で共有リンクを開いた場合は、サインイン後に該当結果画面へ復帰できること
- 共有文面とリンク先では、内部 ID や `Microsoft Entra ID` の実名を自動露出せず、必要に応じてアプリ内表示名のみを用いること

#### 推奨する共通レイアウト

Result 画面は全ゲームで次の `score-first` 構成を共通化する。

1. score hero
2. compact impact summary
3. action area
4. optional detail area

##### 1. score hero

- ゲーム名
- 難易度
- クリア可否
- 自己ベスト更新バッジ
- 今回の記録を最も大きく見せる primary metric

ここでは `今回の記録` を画面内で最も大きく表示し、補助 badge や説明文より先に認識できることを優先する。

##### 2. compact impact summary

- ゲーム別順位変動
- 総合得点の増減
- 総合順位の変動

ここは short label と compact value を中心にし、長い説明 paragraph を初期表示へ並べすぎない。

##### 3. action area

- 再挑戦
- 他ゲームへ移動
- ホームへ戻る
- Microsoft Teams で共有

主要 action は hero score の近くに置き、score を確認した直後に次の行動へ進めることを優先する。

##### 4. optional detail area

- ゲーム固有の補助指標 1 から 2 個
- 自己ベストとの差分
- 状態説明
- share availability や補足説明

例:

- マインスイーパ: クリア時間、ミス回数
- 数独: クリア時間、ヒント使用回数
- Drop Line: line offset、hit rating

この詳細群は初期表示で全文を読ませる前提にせず、compact row または disclosure へ収めてよい。

アクション順は次を推奨する。

1. 再挑戦
2. Microsoft Teams で共有
3. 他ゲームへ移動
4. ホームへ戻る

#### 推奨する Result 画面の見せ方

- 主要数値は画面上部から 1 スクロール以内に収める
- 初期表示では `今回の記録` が画面内で最も大きく見えること
- 重要な変化は色だけでなく、矢印、ラベル、プラスマイナス記号でも表現する
- 説明文は最小限に抑え、補足説明は disclosure や secondary area へ送る
- Microsoft Teams 共有リンクはアクションエリア内に固定し、全ゲームで同じ位置に置く
- モバイルでは縦積み、デスクトップでは 2 段または横並びに再配置する
- 共有リンクはアイコンだけにせず、`Microsoft Teams で共有` の文言を併記する

#### Microsoft Teams 共有文面の推奨テンプレート

`Microsoft Teams` 共有は、短く、競争結果がすぐ伝わり、アプリに戻りたくなる文面を優先する。

##### 推奨テンプレート A

`Arcade で {ゲーム名} {難易度} をプレイしました。今回の結果は {今回の記録}、{自己ベスト更新文}。総合得点 {総合得点}、総合順位 {総合順位} 位です。結果を見る: {結果URL}`

##### 推奨テンプレート B

`{ゲーム名} {難易度} の結果: {今回の記録}。{順位変動文} {総合得点変動文} Arcade で他のゲームにも挑戦する: {結果URL}`

##### テンプレート選定方針

- MVP ではテンプレート A を標準とする
- 自己ベスト更新時は `自己ベスト更新` を優先表示する
- 順位上昇時は順位変動文を優先表示する
- 失敗プレイや未クリア時は過度にネガティブな文言を避け、再挑戦を促す文面へ切り替える
- `中断` または `保存保留` の結果は共有対象外とする

#### Microsoft Teams 共有文面の差し込み値

##### 必須項目

- ゲーム名
- 難易度
- 今回の記録
- 自己ベスト更新有無
- 総合得点
- 総合順位またはゲーム別順位
- 結果画面へ戻れる URL

##### 推奨項目

- 順位変動

- プレイ日時

##### 非推奨項目

- 詳細すぎる計算途中値
- 個人情報性の高い識別子
- `Microsoft Entra ID` の実名やメールアドレス

#### 共有対象の値のおすすめ定義

MVP では、共有対象として次の値を固定採用することを推奨する。

- ゲーム名
- 難易度
- 今回の代表記録 1 件
- 自己ベスト更新有無
- ゲーム別順位変動
- 総合得点
- 総合順位

#### ゲーム別の代表記録のおすすめ

- ゲーム固有の操作ガイドは、少なくとも `run flow`、`board controls`、`finish or abandon` の扱いを含み、ゲームごとの差分だけを差し替えられること

- ランキング説明
- 総合得点の説明
- ヘルプを開閉する導線
- 各画面で使い回せる共通ヘルプ本体

## 5. スコア設計

ゲームが異なると生のスコアの意味が異なるため、ユーザ向けには次の 2 層で扱う。

- ゲーム固有の記録
- 横断比較のための競技ポイント

### 5.1 ゲーム固有の記録

ゲームそのものの魅力を保つため、生の結果はゲームに合った形で保存する。

#### マインスイーパ

- クリア時間
- 難易度
- ミス回数
- クリア可否

#### 数独

- クリア時間
- 難易度
- ミス回数
- ヒント利用回数
- クリア可否

#### Drop Line

- line offset
- 難易度
- hit 成否

### 5.2 競技ポイント

異なるゲーム間で比較可能にするため、各プレイ結果を 0 以上の競技ポイントへ変換する。

#### 基本方針

- ゲーム別ランキングは、生の記録と競技ポイントの両方を見せる
- 総合ランキングは、競技ポイントを用いる
- ユーザにとって理解しやすい計算式にする
- 難易度が高い挑戦ほど高得点を狙いやすくする
- 不正に得点を稼ぎにくい設計にする

#### 初期案

各プレイの競技ポイントは次で算出する。

`競技ポイント = 難易度基礎点 × クリア係数 × 品質係数`

#### 係数案

- 難易度基礎点: Easy, Normal, Hard, Expert のような段階別固定値
- クリア係数: クリアなら 1、未クリアなら 0
- 品質係数: 時間、ミス、ヒント利用を反映する 0 から 1 の係数
- 未クリアまたは `中断` プレイは履歴には残すが、ランキング集計対象外とし、結果画面では `ランキング対象外` と表示する

### 5.3 総合得点

総合得点は、各ゲームの強さを横断的に比較できるように、ゲーム別のベスト競技ポイントを合算して算出する。

#### 初期案

`総合得点(対象期間) = 各ゲームの対象期間内自己ベスト競技ポイントの合計 + 対象期間内の活動ボーナス`

MVP では、総合得点と総合ランキングの対象期間を `シーズン` と `累計` に限定する。

#### 活動ボーナスの考え方

- 毎日遊ぶ動機になる程度に小さく付与する
- 実力差を壊さないよう上限を設ける
- 連続ログインではなく、実際の有効プレイを条件にする

#### この案の意図

- 1 つのゲームだけ極端に強いユーザと、複数ゲームで安定して強いユーザを両立して評価できる
- 新しいゲームが増えても総合得点体系を拡張しやすい

### 5.4 ランキング登録対象

公平性のため、ランキング対象プレイには条件を設ける。

- 正式モードのみ対象とする
- 練習モードはランキング対象外とする
- 異常終了や切断時の扱いを定義する
- ヒント利用の有無によって別ランキングとするか、減点対象とするかを決める
- 未クリアプレイは履歴保存のみ行い、ランキングと総合得点には反映しない
- `中断` プレイは履歴保存のみ行い、ランキングと総合得点には反映しない
- `保存保留` の結果はサーバ受理完了まで集計に反映しない

## 6. 可視化要件

可視化は派手さよりも、ユーザがすぐ状況を判断できることを優先する。

### 6.1 必須の可視化

- 総合得点の推移
- ゲーム別ベスト記録比較
- ゲーム別順位または上位との差分の比較
- 直近プレイの成績推移

### 6.2 推奨チャート

- 時系列推移: 折れ線グラフ
- ゲーム別比較: 棒グラフまたはカード比較
- 順位変動: 折れ線グラフまたは順位推移表
- 総合得点の内訳: 積み上げ棒グラフまたはゲーム別一覧表

### 6.3 表示ルール

- グラフの意味はタイトルと凡例だけで理解できること
- 重要な数値はホバーしなくても見えること
- グラフの近くにテキスト要約を置くこと
- 正確な値を見たいユーザ向けに表形式の代替表示を用意すること
- 色だけで差を表現しないこと

### 6.4 状態別の表示要件

- ローディング中は、データ取得中であることが明確にわかること
- データがまだ存在しない場合は、空状態として次の行動を案内すること
- 通信失敗や保存失敗時は、何が失敗したかと再試行方法がわかること
- ランキング集計中や共有リンク生成中など、待ち時間のある処理では進行状態を伝えること

### 6.5 共通状態ラベル

- `未プレイ`: 対象ゲームまたは対象期間に有効プレイがまだ存在しない状態
- `集計待ち`: 結果は受理されたが、ランキングや総合得点への反映が未完了の状態
- `ランキング対象外`: 練習モード、未クリア、中断などで順位計算の対象外である状態
- `保存保留`: クライアントには結果があるが、サーバ保存が完了していない状態

### 6.6 ホームと詳細画面の情報優先度

ホーム画面では次を優先表示する。

- ゲーム一覧またはグリッド
- 検索、絞り込み、並び替えなどの探索導線
- ヘルプを開く導線
- 初回ユーザや未プレイ user が最初の 1 本を選ぶための強調表示

ホーム画面本文には次の情報を置かないこと。

- 総合得点
- 総合順位
- 各ゲームの最新順位または自己ベスト
- 直近の上昇傾向
- 最近の結果

プロフィールまたは詳細画面では次を主に表示する。

- ゲーム別の強さ比較
- 期間別の成長推移
- 総合得点の内訳
- 自分と上位ユーザとの差分

## 7. ナビゲーション要件

このアプリでは「いま何を遊ぶか」を頻繁に切り替えるため、画面遷移は特に重要である。

### 必須要件

- ログイン後は必ずホーム画面に入り、そのままゲーム選択に入れる
- ホーム画面から任意のゲームへ 1 アクションで入れる
- 各ゲーム画面から他ゲームへ直接移動できる
- 各ゲーム画面からホームへ戻れる
- プレイ結果画面から再挑戦、他ゲーム移動、ホーム復帰のいずれもできる

### ナビゲーションの基本構造案

- ログイン画面
- ホーム兼ゲーム選択画面
- ゲーム画面
- プレイ結果画面
- ランキング画面
- プロフィール画面

### 7.1 ゲーム横断 UI 方針

複数ゲームを切り替えても、ユーザが同じアプリの中にいると直感的にわかるように、ゲーム横断で統一した UI を提供する。

#### 必須要件

- 全ゲームで共通のヘッダ、戻る導線、ゲーム切り替え導線、結果表示ルールを用いる
- 全ゲームで同じ theme system を用いる
- ボタン、カード、ダイアログ、ランキング表示、結果表示の見た目と操作感を統一する
- 結果画面の共有導線は全ゲームで同じ位置と同じ操作感にそろえる
- ゲームごとの差はゲーム盤面やルール説明など、本質的に必要な箇所に限定する

#### テーマ方針

- アプリ全体で統一されたテーマ体系を持ち、ユーザが `Light` と `Dark` から選択できること
- ゲーム盤面を除く UI は、`Light` と `Dark` のどちらでも極力モノトーンの配色に統一し、装飾的な accent color を増やさないこと
- ゲームごとに差を付ける場合も、ゲーム盤面以外では同一テーマの範囲内で最小限の差に留める
- 総合ランキング、ゲーム別ランキング、ホーム、ゲーム画面のすべてで、選択中テーマに応じた同系統の配色ルールを使う
- 色だけに依存せず、ラベル、アイコン、位置関係でも意味が伝わること
- `Light` と `Dark` のどちらでも可読性と状態判別性を維持し、片方だけで成立する配色にしないこと
- header、nav、filter、card などの UI chrome は、装飾的な色面よりも背景トーン差とタイポグラフィで階層を表現すること

### 7.2 情報密度と視覚コミュニケーション方針

文字を読まなくても遊び始められることを優先し、説明文ではなくレイアウト、アイコン、配置、状態変化で意味が伝わる UI を基本とする。

#### 必須要件

- 主要操作、現在地、現在のゲーム状態は、一般的なデスクトップとモバイルの初期表示範囲で把握できるようにし、極力スクロールなしで遊べる構成にする
- 不必要な情報は同時表示せず、折りたたみ、タブ切り替え、セクションの入れ替え、段階表示を使って見た目の単純さを維持する
- ヘッダーは desktop ではできるだけ 1 行で収まり、narrow viewport では hamburger などで折りたためること
- ヘッダーや toolbar では、入れ子の pill、過剰な境界線、重い shadow を多用しないこと
- 画面や機能の分類は、短いラベルまたは monochrome icon で明快に示すこと
- 文章による説明は最小限に留め、ボタン配置、カードの優先度、アイコン、数値の強弱、固定された導線位置で次の行動がわかるようにする
- ヘルプを除き、常時表示する補助説明は短文または単語レベルを基本とし、詳細説明は明示的に開いたときだけ表示する
- 境界線は UI の既定表現にせず、まず余白、整列、背景トーン差でセクションを分けること

#### 補足方針

- ゲーム画面ではプレイ領域を最優先にし、補助情報は盤面を圧迫しない位置へ逃がす
- ゲーム画面では、盤面、進行状態、難易度、離脱導線、必要時の run 終了 action 以外の要素を極力常設しないこと
- ホーム、結果、ランキング、プロフィールでは、意思決定に必要な要素だけを先に見せ、詳細は下位レイヤへ送る
- icon を使う場合も、装飾目的で増やさず、short label で代替できる箇所では label を優先する

## 8. 認証とユーザ管理要件

### 8.1 認証

- 認証は `Microsoft Entra ID` を用いる
- このアプリはスコア登録、ランキング参照、プロフィール保持を伴うため、ブラウザだけで閉じた純粋な静的 SPA ではなく、サーバランタイムを持つ前提で整理する
- 認証後はアプリ内セッションで利用状態を維持する

### 8.2 ユーザプロフィール

- `Microsoft Entra ID` の一意なユーザ識別子を内部ユーザと紐付ける
- 表示名、アイコン、参加日、総合得点、ゲーム別記録、選択テーマを保持する
- ランキング、プロフィール、共有リンク先では、`Microsoft Entra ID` の実名ではなくアプリ内表示名を標準で用いる
- アプリ内表示名は初回ログイン時に Entra の表示名から初期化してよいが、ユーザが後から変更できること
- ランキング上の表示名変更ポリシーを定義する

### 8.3 競争の前提

- 同一ユーザの重複アカウント運用をどこまで許容するか整理する
- MVP は `AzureADMyOrg` を前提に組織内ユーザ向けに始める
- 共有リンク、ランキング詳細、プロフィール閲覧は同一テナントの認証済みユーザに限定する
- 将来のマルチテナント対応は拡張項目として扱う

### 8.4 セッションとサインアウト

- ユーザは明示的にサインアウトできること
- 一定期間内の再訪では、可能な限りログイン状態を維持できること
- セッション切れ時は唐突に失敗させず、再ログインが必要であることを伝えること
- セッション切れがプレイ結果登録に影響する場合は、その結果をユーザに明示すること
- 再ログイン時は、保存済みのユーザ設定として前回選択した `Light` または `Dark` テーマを復元すること

## 12. ユーザ体験上の重要ポイント

- ログイン後に迷わせないこと
- 「いま何を遊ぶか」と「いま自分がどれだけ強いか」がすぐわかること
- ゲーム画面では盤面やプレイ領域を最大限広く取りつつ、操作方法と recovery 情報は Help から即座に開けること
- ゲームごとの操作方法は、そのゲーム画面の primary actions から 1 操作で開け、内容だけ差し替わっても操作体験は変わらないこと
- ゲーム画面では盤面やプレイ領域を最大限広く取りつつ、補助情報は Help や結果画面へ退避して集中を阻害しないこと
- 結果が更新された瞬間に、自己ベスト、順位、総合得点の変化と、その値が暫定か確定かを理解できること
- 複数ゲームをまたいでも同じアプリを使っている一体感があること
- どのゲームへ移動しても UI と monochrome theme system が統一され、学習コストが増えないこと
- 可視化が多すぎて判断しづらくならないこと
- 主要な判断と操作が 1 画面目で済み、不要なスクロールや長文読解を強いないこと
- 不要な情報は折りたたみや入れ替えで退避し、画面を常にシンプルに保つこと
- ヘッダーの分類は、short label または monochrome icon で瞬時に判別できること
- 初回利用時でも、ルール、競争方式、次に取るべき行動が迷わず理解できること
- 通信失敗やセッション切れが起きても、ユーザが状況を理解して復帰できること

## 13. 現時点で決めるべき未確定事項

## 14. 推奨する次の仕様化ステップ

5. シーズン運用ルールと集計バッチの反映タイミングを決める
